# Skill: Client Config

## Purpose

Defines the schema and template for `client.md` — a PM-owned context file that personalizes all agents for a specific client engagement.

Client configs live in **`~/.omp/clients/[name].md`** — inside your PM tooling directory, never inside the client's project repo. You own this file. It travels with you, not with the codebase.

## How agents use client.md

| What agents read | How they adapt |
|---|---|
| `tool` | Use correct field names (Title vs Summary, Tags vs Labels, etc.) |
| `work_item_types` | Use client's names instead of generic ADO/Jira names |
| `methodology` | Use correct sprint/iteration language, story format |
| `estimation` | Use points, t-shirt sizes, or skip sizing |
| `ai_models` | Know what models are available; respect any restrictions |
| `data_restrictions` | Avoid PII, flag sensitive content, use approved tools only |
| `custom_fields` | Include client-required fields in every story output |

## client.md Template

When generating `client.md`, use this format exactly. Save the file as **`~/.omp/clients/[name].md`** where `[name]` is the client name or codename (lowercase, hyphenated — e.g., `acme-corp.md`, `project-falcon.md`).

```markdown
# Client Config
# Generated by client-setup agent. Update as the engagement evolves.

## Engagement
name: [Client or project name / codename]
industry: [Industry or domain]
updated: [YYYY-MM-DD]

## Backlog Tool
tool: [ADO | Jira | Linear | GitHub Issues | Trello | other]
tool_url: [optional — base URL of the board or project]

## Work Item Types
epic: [what they call epics — e.g. "Epic", "Initiative"]
feature: [what they call features — e.g. "Feature", "Theme"]
story: [what they call stories — e.g. "Story", "User Story", "Ticket"]
bug: [what they call bugs — e.g. "Bug", "Defect", "Issue"]
spike: [what they call spikes — e.g. "Spike", "Research", "Enabler"]
task: [what they call tasks — e.g. "Task", "Subtask"]

## Field Names
title_field: [e.g. "Title" in ADO, "Summary" in Jira]
description_field: [e.g. "Description"]
ac_field: [e.g. "Acceptance Criteria" in ADO, part of "Description" in Jira]
tags_field: [e.g. "Tags" in ADO, "Labels" in Jira]
points_field: [e.g. "Story Points", "Estimate", or "none"]

## Custom Required Fields
# List any fields that must be on every story at this client
# Example:
# - Component: [frontend | backend | data]
# - Team: [team name]
custom_fields:
  - [field name]: [allowed values or description]

## Methodology
framework: [Scrum | SAFe | Kanban | Shape Up | Hybrid | other]
iteration_name: [Sprint | PI | Cycle | Week | other]
iteration_length: [1 week | 2 weeks | 4 weeks | other]
estimation: [story points | t-shirt sizes | none]
estimation_scale: [Fibonacci (1,2,3,5,8,13) | XS/S/M/L/XL | other]

## Team
roles: [comma-separated list — e.g. PM, PO, Engineering, QA, Design]
story_author: [who writes stories — PM | PO | collaborative]
ac_reviewer: [who reviews/approves ACs — QA | Engineering | PO | all]

## AI & Tools
approved_models: [Claude | Gemini | ChatGPT | Copilot | none specified]
data_restrictions: [none | no PII | no external AI | other]
other_tools: [Confluence | Notion | Miro | Productboard | other | none]

## Notes
# Free text — anything that shapes how you work with this team
notes: |
  [Optional notes about team culture, preferences, or constraints]
```

## Loading behavior

Agents look for client config in this order:

1. **`~/.omp/clients/[name].md`** — check this directory for available client configs
   - If exactly one file exists, load it automatically
   - If multiple files exist, ask the user: "Which client are we working on?" and list the options
   - If no files exist, suggest running `/setup` to create one

2. **Do not look in the project directory.** Client configs are never stored in the project repo.

## Switching clients

To switch engagements, tell the agent: *"Switch to [client name]"* or run `/setup` to create a new config.

## Versioning

`client.md` is yours — it lives in `~/.omp/clients/` and evolves as the engagement evolves. Update it when the team changes tools, methodology, or adds custom fields. Never commit it to the client's project repo.
